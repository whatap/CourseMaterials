# 50 – WhaTap 대시보드 읽기

> **언어:** [English](50-whatap-dashboard.md) | 한국어

**이 챕터에서 배울 내용:**
- 어느 메뉴를 먼저 열어야 하는가 (그리고 어느 메뉴는 나중으로 미룰까)
- **Hitmap** 읽는 법 — 가장 자주 보게 될 화면
- 트레이스 트리에서 느린 스팬 찾는 법
- Apdex와 응답시간 히스토그램이 말해주는 것

> **언제 이 챕터로 와야 하나?** 40챕터를 마치고 `make k6-start`(또는 `./stress-test.sh`)로 트래픽이 흘러서 대시보드에 점이 찍히기 시작했을 때.

---

## 주요 메뉴 한눈에

| 메뉴 | 본다 |
|------|------|
| 대시보드 / 개요 | 처리량(TPS), 평균 응답시간, 에이전트 연결 상태 |
| 트랜잭션 / Hitmap | 모든 요청을 점으로 — X = 시간, Y = 응답시간 |
| 프로파일 / 트레이스 | 한 요청 안에서 어떤 메서드/서비스가, 얼마나 오래 호출됐는지 |
| 로그 | 에이전트가 수집한 애플리케이션 로그 |
| 토폴로지 | 서비스 간 호출 관계 그래프 |

이 실습에서는 **Hitmap부터 시작하세요.** 나머지 메뉴는 거기서 다 따라 들어갈 수 있습니다.

## Hitmap 읽는 법

Hitmap은 산점도(scatter plot)입니다.

- **점 1개 = 완료된 요청 1개.**
- **X축 = 시간** (요청이 끝난 시각).
- **Y축 = 응답시간**(ms). 위로 갈수록 느림.
- **색 = 상태** (정상/에러/느림).
- **박스 드래그** — 그 영역의 점만 선택해 아래쪽 트레이스 목록을 필터링.

**느린 요청 찾는 레시피:** *오른쪽 위*(최근 + 느림) 영역에 박스를 드래그한 다음, 점 하나를 클릭해 트레이스를 엽니다.

→ 참고: [WhaTap — Hitmap](https://docs.whatap.io/ko/java/hit-map)

## 트레이스 트리 읽는 법

트레이스는 **스팬(span)**의 트리입니다. 스팬 1개 = 작업 1단위 (HTTP 호출, DB 쿼리, 메서드 호출 등).

- **맨 위 스팬**은 진입점 (여기서는 `gateway`).
- **들여쓰기** = 호출 깊이.
- **막대 길이** = 그 단계에서 머무른 전체 시간.
- **막대 — 자식 막대들** = 그 단계 자체에서 든 시간 (호출한 곳에서 든 시간 빼고).

이 실습에서 레이턴시는 보통 가장 깊은 잎이 만듭니다 — 대부분 최대 3초까지 sleep하는 `inventory`. 매 레벨마다 가장 긴 막대를 따라 내려가면 닿게 됩니다. 같은 흐름의 가이드 실습은 [실습 4](80-exercises.ko.md#실습-4-inventory-서비스--트레이스-깊이) 참고.

→ 참고: [WhaTap — 트랜잭션 프로파일](https://docs.whatap.io/ko/java/trs-profile)

## Apdex와 응답시간 히스토그램

**Apdex**는 0–1 사이의 사용자 만족도 점수입니다. 목표 응답시간 `T`(`whatap.conf`의 `apdex_time` 설정)에서 파생됩니다:

| 구간 | 범위 | 가중치 |
|------|------|--------|
| 만족 (Satisfied) | `≤ T` | 1.0 |
| 허용 (Tolerating) | `T < x ≤ 4T` | 0.5 |
| 불만족 (Frustrated) | `> 4T` | 0.0 |

이 실습은 `apdex_time=2000`이므로 2초 이하는 만족, 8초 초과는 불만족입니다. `inventory`가 최대 3초까지 sleep하므로 **허용** 구간이 의미 있게 잡힙니다.

→ 참고: [Apdex — apdex.org](https://www.apdex.org/overview.html)

## 에이전트 이름 보기 (compose-1 vs compose-2)

같은 서비스라도 어느 compose 파일로 띄웠느냐에 따라 대시보드에 매우 다른 이름으로 나타납니다:

| 시작 파일 | 대시보드에 표시되는 에이전트 이름 |
|-----------|----------------------------------|
| `docker-compose-whatap-1.yml` | 자동 생성 — 인스턴스 구분 어려움 |
| `docker-compose-whatap-2.yml` | `product-composite-{ip2}-{ip3}` 등 — 인스턴스가 명확히 구분됨 |

**compose-2**에서 `product-composite`를 5개로 스케일하면 5개의 깔끔하게 이름 붙은 에이전트를 얻습니다 — [실습 1](80-exercises.ko.md#실습-1-에이전트-이름-설정)의 핵심.

## 첫 점검 체크리스트

대시보드를 열고 다음을 확인합니다:

- [ ] **에이전트 6개 이상 Connected** (gateway, product-composite, product, recommendation, review, inventory — 그리고 스케일된 인스턴스 추가).
- [ ] **Hitmap에 점이 가로로 연속 분포** — 트래픽이 들어오는 중.
- [ ] **가장 위쪽 점 클릭** → 4-홉 트리(gateway → product-composite → product → inventory)가 펼쳐짐.

하나라도 안 되면 [90 – 트러블슈팅](90-troubleshooting.ko.md)으로.
